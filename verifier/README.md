# verifier/

Training, evaluation, and corpus tooling for the verifier-gated cascade router (IP-1).

This directory holds the Day-1 OAuth-judge validation harness. The verifier model training scripts (`train.py`, `eval.py`, `corpus_builder.py`, `quantize.py`) land in later weeks per the IP-1 blueprint.

---

## FOUNDER APPROVAL GATE — read this first

The validation runner makes real calls against the founder's Claude Max subscription via OAuth. Before any such call:

1. The founder must confirm that Anthropic's TOS allows this usage pattern (subscription token used for programmatic batch labeling against `/v1/messages`).
2. The schema migration `backend/migrations/verifier_training_corpus_nullable_label.sql` must be applied (via Supabase MCP or the dashboard).
3. The env var `FOUNDER_APPROVED=1` must be set in the runner's shell. The runner raises `RuntimeError` at function entry without it (when `supabase_client` is not explicitly injected).

Do NOT add `FOUNDER_APPROVED=1` to any committed `.env` file. The gate exists to force a conscious, per-invocation decision.

## Running the validation slice

### Prerequisites

- OAuth login completed (e.g. `nadirclaw auth anthropic login` or equivalent), so `get_credential("anthropic")` returns a token starting with `sk-ant-oat`.
- Schema migration applied (see above).
- `verifier/requirements.txt` installed: `pip install -r verifier/requirements.txt`. Uncomment `nadirclaw>=0.13.0` for the real run.
- Supabase service-role env vars: `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`.

### Step 1 — Seed the 5 hand-labeled validation triples

```bash
# Dry-run first (default; prints what would be inserted, no DB write):
python verifier/seed_validation_triples.py

# Inspect the output. Then, with founder approval, actually insert:
FOUNDER_APPROVED=1 python verifier/seed_validation_triples.py --no-dry-run
```

This writes `verifier/seed_labels.json` mapping each row's UUID to its expected human label (1, 0, 0, 1, null).

### Step 2 — Run the slice

```bash
FOUNDER_APPROVED=1 python verifier/validation_runner.py --n 50
```

The runner:
- Queries `verifier_training_corpus WHERE label IS NULL AND label_source='pending_oauth_judge' ORDER BY created_at LIMIT n`.
- Sends each triple through `OAuthJudgeClient` (Claude Sonnet via subscription OAuth).
- Upserts each successful label back to the row immediately (no batch).
- Triggers a kill switch after 3 consecutive errored calls and writes a partial report.
- Writes `verifier/reports/validation_slice_<UTC timestamp>.json`.

### Step 3 — Interpret the report

Report keys:

| Key | Meaning |
|---|---|
| `n_requested`, `n_judged`, `n_errors` | Counts. `n_judged` = labels persisted. |
| `label_distribution` | `{"0": n_reject, "1": n_accept}`. Skew check. |
| `avg_confidence` | Mean judge-reported confidence on labeled rows. |
| `wall_time_seconds` | End-to-end runtime. |
| `estimated_full_batch_seconds` | Linear extrapolation to 10K triples. Lower bound. |
| `calibration.agreed / total` | Agreement on the 4 clear seed triples. 4/4 = pass. |
| `calibration.observed_borderline` | Judge's call on triple 5 (recorded, not scored). |
| `calibration.disagreements` | Per-row breakdown of any miss. |
| `abort_reason` | Non-null means kill-switch tripped; rest of report is partial. |
| `random_examples` | 5 random judged rows for spot-checking. |

### Step 4 — Calibration verdict

- **4/4 agreement** on the clear triples → calibration passes; the prompt template is acceptable. Plan the next slice with the founder before scaling.
- **<4/4 agreement** → `calibration_concern=true`. Inspect `disagreements` and the judge's `rationale`. Refine `judge_prompt_template.txt` and re-run the slice. Do NOT proceed to a larger batch.
- **3 consecutive errors** (`abort_reason` set) → check OAuth token freshness, rate-limit status, and the partial report's last `error` field before retrying.

### Do NOT scale beyond 100 triples without founder sign-off

The runner enforces `n <= 100` via a synchronous `ValueError`. Any larger pass requires:
1. A passing 4/4 calibration result on a slice.
2. Explicit founder approval to spend more of the Claude Max 5-hour message budget.
3. Re-running the slice with a fresh `FOUNDER_APPROVED=1` invocation.

| File | Purpose |
|---|---|
| `oauth_judge.py` | `OAuthJudgeClient` + `JudgeResult`. Injectable credential / transport / sleep. |

## Local training

`verifier/train_local.py` is the standalone terminal equivalent of
`verifier/colab_train.ipynb`. It trains the DeBERTa-v3-small
cross-encoder verifier on the JSONL corpus at
`verifier/data/routerbench_triples.jsonl` and writes a best checkpoint
plus an INT8 dynamic-quantized CPU artifact to `verifier/weights/`.

### Install

```bash
pip install -r verifier/requirements_local.txt
```

These deps are intentionally separate from the backend's
`requirements.txt`. Do not install them into the production image.

### Run

```bash
python verifier/train_local.py \
    --data verifier/data/routerbench_triples.jsonl \
    --output verifier/weights/
```

The script auto-detects the best available device in this priority
order: CUDA, then Apple Silicon MPS, then CPU. Override with
`--device cpu|mps|cuda`. Pass `--max-rows 500` for a smoke run.

### Realistic timings (90K train split, 4 epochs, batch 8, max length 512)

| Device | Wall time | Notes |
|---|---|---|
| Local CUDA GPU (T4 / 3090 / 4090 class) | approx 30 minutes | fp16 enabled automatically |
| Apple Silicon MPS (M2 / M3 / M4) | approx 2 to 4 hours | fp32, no fp16 on MPS |
| CPU (modern x86 / arm64) | approx 12 to 24 hours | not recommended outside smoke runs |

### Output

```
verifier/weights/
  best/                    # HuggingFace checkpoint (pytorch_model.bin or safetensors) + tokenizer
  verifier_int8.pt         # INT8 dynamic-quantized CPU weights for serving
  checkpoints/             # per-epoch trainer checkpoints (save_total_limit=2)
```

### Quality gate

The script prints AUROC, precision, recall, F1, and p50 CPU inference
latency at the end. Per the IP-1 blueprint, the production cascade
requires **AUROC >= 0.82** on the held-out test split. If the run falls
below the gate, the script prints a loud `DO NOT DEPLOY` warning and
still saves the checkpoint so it can be inspected. Do not ship a
sub-gate checkpoint to the cascade router.

### Smoke test

```bash
cd backend
python -m pytest tests/test_train_local_smoke.py -v
```

The smoke test mocks transformers / datasets and exercises only the
argparse, JSONL loader, tokenization-format helper, and split-assignment
logic. No model download, no training, no GPU required.
