Three Ways to Cut Your LLM API Costs: Routing, Context Compression, and Prompt Caching

June 20, 2026

Most engineering teams discover the same thing about six weeks after going to production with an LLM-backed product: the bill is much larger than the prototype suggested. Not because usage grew faster than expected (though that happens too), but because the underlying API spend was never optimized in the first place.

The naive path is to call the best available model on every request and hope the product finds product-market fit before the cloud bill becomes a boardroom conversation. That works at prototype scale. It falls apart at production scale, where the same structural waste compounds across thousands or millions of requests per day.

There are three independent strategies that can eliminate most of that waste. Each addresses a different cost driver. Each can be applied without changing your application logic. And critically, they stack: getting all three right does not just add their savings — it multiplies them.

Where the money actually goes

Before optimizing, it helps to understand what you are paying for. LLM API costs have two components: input tokens (everything sent to the model) and output tokens (everything the model generates). Output tokens cost roughly 3-5x more per token than input on most providers, but input usually dominates in volume for agent-heavy and tool-heavy workloads.

In a typical production LLM application, input token costs break down roughly like this:

Token source Share of input tokens Optimization lever
System prompt (repeated every request) 15–40% Prompt caching
Tool / function schemas 10–30% Context compression
Conversation history 20–50% Context compression
Retrieved context (RAG chunks) 10–40% Context compression + caching
Actual user prompt (new content) 5–20% Model routing

The user's actual prompt is often the smallest component. The overhead — schemas, history, system instructions, retrieved documents — is where most of the tokens go. Any cost strategy that only looks at the user prompt is missing the vast majority of the problem.

Strategy 1: Intelligent model routing

The first and most impactful strategy is routing requests to the cheapest model that can handle them correctly. This does not mean sending every request to the cheapest model — it means classifying each request by complexity and using appropriately priced models for each tier.

The economics are striking. Claude Opus 4.6 costs $15 per million input tokens. Claude Haiku costs roughly $0.80 per million. That is a 19x price gap. For a request like "what files are in the src/ directory?", Haiku and Opus produce the same answer. Routing the simple request to Haiku and reserving Opus for genuinely hard tasks (multi-file reasoning, architectural analysis, complex debugging) is where most of the savings come from.

In practice, workloads that mix simple and complex requests — which is nearly all production LLM applications — see 30-60% cost reductions from routing alone. A coding assistant session might make 200 API calls, of which 130 are simple file reads, directory lookups, and status checks that genuinely do not require a frontier model.

The classifier matters: Routing only works if the classifier is accurate. A false positive — sending a hard prompt to a cheap model — produces a wrong or incomplete answer and potentially costs more overall when the user has to retry. Good classifiers err on the side of routing up, not down.

Strategy 2: Context compression

Routing optimizes which model receives a request. Context compression optimizes what gets sent to the model at all. These are independent problems — a request routed to the cheapest model can still carry vastly more tokens than necessary.

The structural sources of token waste in LLM payloads are well understood:

  • Pretty-printed JSON: Indentation and newlines make JSON readable to humans but add 20-40% overhead versus compact representation. The model does not need the whitespace.
  • Repeated tool schemas: Agent frameworks often include the full schema for every tool on every turn of the loop. If a workflow has 10 tools and runs 30 turns, each schema is sent 29 unnecessary times.
  • Stale conversation history: A 60-turn debugging session carries turns 1-40 in the context even after those turns are no longer relevant to the current task.
  • Duplicated system prompts: Some orchestration frameworks embed the system prompt in user messages, doubling the cost of the most expensive recurring token block.
  • Whitespace bloat: Multiple blank lines and trailing spaces in pasted code, logs, or documentation add tokens with zero information content.

Lossless transforms — JSON minification, schema deduplication, whitespace normalization, history trimming — typically cut input token counts by 30-70% on payloads where these patterns are present. The savings are largest on agentic workflows (where schemas repeat across many turns) and on structured data analysis (where JSON formatting dominates token counts).

Beyond lossless transforms, semantic deduplication can identify near-duplicate messages across a conversation and replace redundant content with compact references while preserving meaningful differences. This requires care: a 90%-similar follow-up message might contain the 10% that changes everything about the task. Naive semantic dedup that collapses those refinements produces wrong outputs silently, which is worse than wasting tokens.

The right implementation extracts and preserves the unique differences before compacting the shared content. Nadir's Context Optimize feature does this with word-level diff extraction, backed by 60 automated accuracy tests.

Strategy 3: Prompt caching

Prompt caching is a provider-level feature that both Anthropic and OpenAI now offer. It allows you to mark specific prefixes in your prompt as cacheable — and pay a heavily discounted rate when that prefix is reused across subsequent requests within the cache TTL.

Anthropic's implementation charges 25% of the normal input rate for cache hits, with a 5-minute TTL (extendable to 1 hour for larger prompts). OpenAI's implementation gives cache hits for free — zero input token cost for the cached prefix.

The mechanics work like this: you mark a portion of your prompt (typically the system prompt, a large document, or a set of tool schemas) with a cache control marker. On the first request, you pay the full input rate and the provider stores the KV cache for that prefix. On subsequent requests that share the same prefix, you pay the cache write or hit rate instead.

# Anthropic: mark the system prompt for caching
{
  "model": "claude-opus-4-6",
  "system": [
    {
      "type": "text",
      "text": "You are a senior software engineer...[long system prompt]",
      "cache_control": {"type": "ephemeral"}
    }
  ],
  "messages": [...]
}

# Cache hit on second request with same system prefix:
# cost = 0.25 × normal_input_rate instead of 1.0 × normal_input_rate

The savings from caching depend heavily on how much of your input is stable across requests. A customer support bot with a 10,000-token system prompt that gets reused across every conversation sees enormous cache value. A code generation tool with a long, stable context document sees the same benefit. Workloads where every token is unique (summarization of fresh documents, one-shot analysis tasks) see no cache benefit.

When caching helps most

Workload type Stable prefix % Cache benefit
Customer support (fixed system prompt) 60–80% High
Code assistant (stable codebase context) 40–70% High
Agentic loop (repeated tool schemas) 20–50% Medium
RAG pipeline (rotating retrieved chunks) 10–30% Low–Medium
Document summarization (fresh input each time) <10% Low

One practical note: caching requires that the cached prefix is byte-for-byte identical across requests. Templating engines that inject dynamic content early in the system prompt (timestamps, user names, session IDs) break the cache on every request. The fix is to move all dynamic content to the end of the prompt, after the cacheable static section. This is a structural change that pays dividends even if you later switch providers.

Stacking all three: the math

What happens when all three strategies are applied together? The savings compound rather than add.

Take a concrete example: a production AI coding assistant running at 1 million requests per month, averaging 4,000 input tokens per request, primarily on Claude Opus 4.6 at $15/M tokens. Baseline monthly input cost: $60,000.

Optimization layer Mechanism Tokens remaining Monthly cost Saved vs. baseline
Baseline No optimization 4,000/req $60,000 —
+ Context compression Lossless transforms (safe mode) ~2,000/req $30,000 50%
+ Prompt caching 1,200 stable tokens cached ~2,000/req (900 billed at 25%) $16,650 72%
+ Model routing 60% of requests on Haiku Same tokens, cheaper models ~$7,800 87%

An 87% reduction on a $60,000 monthly bill is $52,200 per month in savings. The three strategies together achieve what none achieves alone. Routing saves money by choosing the right model. Compression saves money by reducing what is sent to the model. Caching saves money by avoiding re-billing for content the provider already processed.

Why most teams only implement one

In practice, teams usually implement exactly one of these strategies — whichever one their framework makes easiest. OpenAI's Python SDK makes prompt caching automatic when the conditions are met. Anthropic's documentation leads with system prompt caching. Neither provider's documentation tells you to also compress your context and route your requests.

The three strategies are also owned by different parts of the stack. Routing is typically a proxy layer concern. Compression is a preprocessing concern. Caching is a prompt engineering and API request construction concern. Getting all three requires someone to own the full picture, not just the piece closest to their current task.

Measuring the results

The only way to know what a given optimization actually saves on your workload is to measure it. Synthetic benchmarks (including our own) give directional guidance but cannot predict your exact numbers because your token distribution, prompt structure, and request mix are unique.

For routing, the key metric is the distribution of classified tiers. If 80% of your requests are classified Simple, routing saves a lot. If 80% are classified Complex or Reasoning, the savings are smaller but quality is fully preserved.

For compression, measure the before and after token counts on a sample of real production payloads. Nadir's dry-run mode does this without touching any live request:

# Analyze a captured payload without touching production
nadirclaw optimize payload.json --mode safe --format json

# Output includes per-transform breakdown:
# {
#   "original_tokens": 3657,
#   "optimized_tokens": 1573,
#   "tokens_saved": 2084,
#   "savings_pct": 57.0,
#   "transforms_applied": ["json_minify", "tool_dedup", "whitespace_norm"]
# }

For prompt caching, your provider's usage dashboard will show cache hit rates once cache control markers are in place. Aim for 50%+ hit rates on the cacheable prefix; below that, the time and engineering cost of structuring your prompts for caching may not pay off.

Where to start

If you are running LLM-backed features in production and have not optimized your API spend, the order of operations that tends to pay off fastest is:

  1. Compression first. It requires no changes to your prompt design or application logic. Install Nadir, enable safe mode, and capture real savings immediately. This gives you a measurement baseline for the other strategies.
  2. Routing second. Enabled by default when you run nadirclaw serve. Review the tier distribution in your logs to confirm it matches your intuition about your workload complexity mix.
  3. Caching third. Audit your system prompt and context structure. Move dynamic content to the end. Add cache_control markers to stable prefixes. Monitor hit rates over the following week.

Realistic expectations: Teams that implement all three strategies consistently see 60-85% reductions in input token spend. The wide range reflects real workload differences — a customer support bot with a stable 8,000-token system prompt hitting the same models all day sees different numbers than a code analysis pipeline with varied, fresh inputs. What matters is measuring your specific workload, not benchmarking against averages.

The underlying economics of LLM APIs are not going away. Model providers compete on capability, and the most capable models command the highest prices. The teams that build sustainable LLM products are the ones that develop infrastructure to use those models selectively — expensive where the task demands it, cheap where it does not.
