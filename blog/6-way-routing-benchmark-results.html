How We Benchmarked Our 6-Way LLM Routing System

March 22, 2026

LLM routing promises significant cost savings, but promises are cheap. We wanted hard numbers. So we designed a controlled benchmark that tests every combination of our two cost-saving features — tier-based routing and context optimization — against a pure baseline of sending everything to the most expensive model.

The short version: the best configuration saved 48% on costs, with quality verified by an independent LLM judge and no significant degradation detected. Here is how we ran the experiment and what we learned.

The 6 configurations

We tested six variants, each progressively layering on cost-saving features:

  1. V1 — Baseline (Opus): Every prompt goes to Claude Opus 4.6. No routing, no optimization. This is the control.
  2. V2 — Router Only: NadirClaw's heuristic classifier analyzes each prompt and routes simple ones to cheaper models (Haiku, Sonnet) while keeping complex prompts on Opus.
  3. V3 — Safe Optimize Only: No routing. All prompts go to Opus, but input tokens are compacted using lossless transforms (JSON minification, whitespace normalization, tool schema dedup).
  4. V4 — Router + Safe Optimize: Both features combined. Simple prompts get routed to cheaper models and their input tokens get compacted.
  5. V5 — Aggressive Optimize Only: No routing. All prompts go to Opus, but input tokens are compacted using both lossless transforms and semantic deduplication via sentence embeddings.
  6. V6 — Router + Aggressive Optimize: Full stack. Routing plus aggressive context compaction.

The prompts

We assembled a diverse set of real-world prompts spanning simple factual questions to complex system design tasks. The prompts fell into three natural categories:

  • Simple — factual questions, status checks, format conversions. The kind of work where a smaller model is perfectly capable.
  • Medium — code generation, explanations with nuance, multi-step reasoning. These benefit from a capable model but do not require the most expensive tier.
  • Complex — system architecture design, multi-file refactoring plans, nuanced technical analysis. These genuinely need a frontier model.

This distribution mirrors what we see in real production traffic: a mix of trivial and hard requests, with the majority falling somewhere in the middle.

Methodology

Every configuration was tested under identical conditions:

  • Environment: Claude CLI for consistent request handling, eliminating variability from different HTTP clients or SDK versions.
  • Quality scoring: Each response was evaluated by an independent LLM-as-judge (Claude Sonnet 4.6) that compared the routed response against the baseline Opus response. The judge scored on correctness, completeness, and clarity.
  • Cost measurement: Actual token counts from the API response, priced at each model's published rates. No estimates or approximations.
  • Latency: Wall-clock time from request to last token, measured by the CLI.

Using an LLM as judge is not perfect, but it scales and removes human bias. We chose Sonnet specifically because it is capable enough to evaluate quality but is not the model being tested, avoiding circular evaluation.

Results: cost per request

Configuration Avg cost/req Savings
V1 Baseline (Opus) $0.00849 —
V2 Router Only $0.00592 30%
V3 Safe Optimize $0.00625 26%
V4 Router + Safe $0.00458 46%
V5 Aggressive Optimize $0.00612 28%
V6 Router + Aggressive $0.00444 48%

The standout finding: routing and optimization are complementary, not redundant. V2 (routing alone) saves 30%. V3 (optimize alone) saves 26%. But V4 (both together) saves 46% — nearly the sum of the two, because they reduce cost on different axes. Routing picks a cheaper model. Optimization reduces the input tokens sent to whatever model is chosen.

Aggressive optimization (V5, V6) added only a small incremental gain over safe optimization (V3, V4). The extra 2% from semantic dedup is real but modest for most workloads. Safe mode is the better default for production because it is deterministic, lossless, and has zero ML dependencies.

Results: per-category savings

The savings are not uniform across prompt types. Breaking down V6 (Router + Aggressive) by category:

Category Savings vs baseline
Simple prompts ~80%
Medium prompts ~40%
Complex prompts ~49%

Simple prompts saved the most because they get routed to Haiku, which is roughly 10x cheaper than Opus. The router correctly identifies these as not needing a frontier model.

Medium prompts saw a moderate ~40% savings. These typically get routed to Sonnet, which costs about 3x less than Opus.

Complex prompts still saved ~49%, primarily from context optimization rather than routing. Since the router correctly keeps these on Opus, the savings come entirely from reducing input token waste — minified JSON, deduplicated tool schemas, and trimmed chat history.

Quality verification

Quality was verified by an independent LLM-as-judge (Claude Sonnet 4.6) with no significant degradation detected across any configuration or savings level. The judge compared each routed response against the baseline Opus response on correctness, completeness, and clarity.

This result makes sense when you consider what the router is doing. It is not downgrading hard tasks. Simple factual questions get the same correct answer from Haiku as from Opus. The router only sends prompts to cheaper models when the complexity analysis indicates they do not require frontier-level reasoning.

Latency

An unexpected benefit: routed requests were significantly faster. The average response time dropped from 536ms (Opus baseline) to 208ms for the routed configurations. Cheaper models tend to be faster models, so routing saves time in addition to money.

The heuristic classifier

NadirClaw's routing classifier is deliberately simple. It is a rule-based heuristic system, not a neural network. It analyzes prompt length, vocabulary complexity, presence of code or structured data, and a handful of other surface-level features to assign a complexity tier.

Why heuristic instead of ML? Three reasons:

  • Latency: Classification takes less than 1ms. An ML model would add 10-50ms of overhead, eating into the latency gains from routing.
  • Dependencies: Zero. No PyTorch, no model files, no GPU. The classifier is pure Python with no imports beyond the standard library.
  • Debuggability: When a prompt gets misrouted, you can trace exactly which rule fired and why. With an ML model, debugging misclassifications requires inspecting embeddings and attention weights.

The heuristic approach means the classifier is not perfect. It will occasionally route a medium-complexity prompt to Haiku when Sonnet would have been better, or keep a simple prompt on Opus when Haiku would suffice. But the benchmark shows that even with these edge cases, the aggregate savings are substantial.

ROI at scale

At $10,000/month in LLM spend, the 48% savings from V6 translates to roughly $4,800/month saved (before routing fees). After NadirClaw's routing fee, the net savings are approximately $3,600/month, or $43,200/year.

At $50,000/month, net annual savings reach approximately $215,000.

The math scales linearly. The classifier runs locally at negligible cost, so the only variable cost is the routing fee, which is a percentage of savings — you only pay when you save.

How to enable routing

NadirClaw is a drop-in proxy. Point your base URL at it and add your API key:

# Install
pip install nadirclaw

# Start the router with safe context optimization
nadirclaw serve --optimize safe

Then configure your client to use the local proxy:

# Claude Code
claude config set --global apiBaseUrl http://localhost:8856/v1

# Or via environment variable for any OpenAI-compatible client
export OPENAI_BASE_URL=http://localhost:8856/v1

For per-request control, you can pass routing parameters in the JSON body:

curl http://localhost:8856/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-API-Key: your-key" \
  -d '{
    "model": "auto",
    "optimize": "safe",
    "messages": [{"role": "user", "content": "What is 2+2?"}]
  }'

Setting model to "auto" enables routing. The router will classify the prompt and select the appropriate model tier. You can also set model to a specific model name to bypass routing for individual requests.

What we learned

Three takeaways from running this benchmark:

  1. Medium-tier models are the sweet spot. Sonnet-class models handle the vast majority of real-world prompts at a fraction of the cost. The frontier models (Opus, GPT-4) are genuinely needed only for complex reasoning and multi-step analysis. A good router exploits this gap.
  2. Routing and optimization are complementary. They reduce cost on orthogonal axes — routing picks a cheaper model, optimization reduces token count. Stacking them approaches the theoretical maximum savings without sacrificing quality.
  3. Simple heuristics work surprisingly well. Our rule-based classifier, running in under 1ms with zero dependencies, captured the majority of the savings that a more complex ML system would provide. Complexity in the classifier does not proportionally improve routing accuracy for most workloads.
