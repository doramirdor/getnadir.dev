How Nadir Saves 30-60% on LLM API Costs

March 10, 2026

We built Nadir because we kept watching our LLM bills climb while knowing most of those API calls were doing trivial work. A Claude Code session might make 200 API calls in an hour. Maybe 30 of those are genuinely hard: writing a new module, debugging a race condition, reasoning about system architecture. The other 170? Reading files, listing directories, checking git status, reformatting output. Every single one of those was hitting Claude Opus at $15 per million input tokens.

That's the problem. Not that LLMs are expensive, but that we treat every request like it needs the most powerful model available.

The price gap is enormous

Look at the actual pricing across Anthropic's lineup (as of early 2026):

  • Claude Opus 4.6: $15/M input, $75/M output
  • Claude Sonnet 4.6: $3/M input, $15/M output
  • Claude Haiku 3.5: $0.80/M input, $4/M output

Haiku costs roughly 5% of what Opus costs for input tokens. For a prompt like "list the files in the src/ directory," Haiku produces identical output. You are paying a 20x premium for zero additional value on that request.

The same gap exists on the OpenAI side. GPT-4o runs $2.50/M input vs. GPT-4o-mini at $0.15/M. Google's Gemini lineup shows similar spreads. The model providers themselves are telling you: not every task needs the biggest model.

How the classifier works

Nadir sits between your application and the LLM provider as a local proxy. Every incoming request passes through a complexity classifier before it reaches any model.

The classifier is a lightweight heuristic system (not a neural network) that analyzes surface-level features of the prompt: token count, vocabulary complexity, presence of code blocks or structured data, whether the prompt asks for generation vs. retrieval, and a handful of other signals. Based on these features, it assigns the prompt to one of four tiers:

  • Simple: Factual lookups, file reads, format conversions, status checks. Routes to Haiku or GPT-4o-mini.
  • Mid: Code explanations, moderate generation tasks, straightforward debugging. Routes to Sonnet or GPT-4o.
  • Complex: Multi-file code generation, nuanced technical analysis, system design. Stays on Opus or GPT-4.
  • Reasoning: Multi-step logical reasoning, architectural decisions, complex refactoring plans. Stays on the most capable model available.

Classification takes under 1ms. Compare that to LLM response times of 500ms-30s. The overhead is invisible.

Why heuristics instead of ML?

We considered training a neural classifier and actually built one (a gradient-boosted tree model that ships with the hosted version). But for the open-source release, we went with heuristics for three reasons.

First, latency. A rule-based classifier runs in microseconds. An ML model adds 5-50ms of overhead, which matters when you are proxying thousands of requests.

Second, zero dependencies. The heuristic classifier is pure Python with no imports beyond the standard library. No PyTorch, no model weights, no GPU. You install it and it works.

Third, debuggability. When a prompt gets misrouted (and it will, sometimes), you can trace exactly which rule fired and why. With an ML model, you get a probability score and a shrug.

A concrete example

Here is a real breakdown from a 45-minute Claude Code session refactoring a FastAPI backend. 187 total API calls:

  • 112 calls (60%) classified Simple: file reads, directory listings, git status, grep searches, config checks. Routed to Haiku. Average cost per call: ~$0.0004
  • 48 calls (26%) classified Mid: code explanations, docstring generation, test writing, simple bug fixes. Routed to Sonnet. Average cost per call: ~$0.003
  • 27 calls (14%) classified Complex/Reasoning: new module creation, architectural reasoning, complex refactoring. Stayed on Opus. Average cost per call: ~$0.02

Without routing, all 187 calls on Opus: roughly $3.74. With routing: roughly $1.27. That is a 66% reduction on this particular session. Across many sessions, we consistently see 30-60% savings depending on the workload mix.

What about quality?

The obvious concern: does routing to cheaper models hurt output quality? For the simple and mid tiers, we have not seen meaningful quality differences. Haiku does not produce worse output when you ask it to read a file. Sonnet writes perfectly good docstrings.

The key is that complex and reasoning-tier prompts never get downgraded. The classifier is intentionally conservative. When it is unsure, it keeps the request on the more capable model. A false positive (sending a hard prompt to a cheap model) is much worse than a false negative (sending an easy prompt to an expensive model). The classifier errs on the side of quality preservation.

In our 30-prompt benchmark, an independent LLM judge confirmed 87% quality preservation across all tiers. The 13% that showed differences were edge cases at category boundaries, not catastrophic failures.

Getting started

Setup is two commands:

# Install
pip install nadirclaw

# Start the router
nadirclaw serve

Then point your LLM client's base URL to http://localhost:8856/v1. For Claude Code specifically:

claude config set --global apiBaseUrl http://localhost:8856/v1

That's it. Your existing code, prompts, and workflows stay exactly the same. The only thing that changes is your bill.
