The agentic token spiral: why AI agent costs scale faster than usage

A token-level analysis of where costs accumulate in ReAct agent loops, and a practical framework for reducing them without sacrificing output quality.

June 30, 2026

Abstract. Single-prompt LLM calls have a predictable cost structure: you send a prompt, you receive a response, you pay for both. Agentic workflows break this model. In a ReAct-style agent loop, each additional turn re-sends the system prompt, all tool schemas, accumulated conversation history, and freshly retrieved context. Token consumption scales with the square of loop depth, not linearly with task complexity. This analysis models a typical 10-turn research agent to quantify waste by stage, compare four optimization strategies across cost and quality dimensions, and propose a seven-step instrumentation framework. The main finding: in an unoptimized agent, over 60% of input tokens are structural overhead that does not change between turns. Eliminating this waste through routing, compression, and prompt caching reduces estimated monthly cost by up to 79% on illustrative workloads, with less than 5 percentage points of quality impact when done correctly.

Introduction: when agents meet the API bill

The first LLM prototype you build is cheap. One prompt, one response, $0.002 per interaction. The production agent you deploy six months later is not cheap. It makes five to fifteen API calls per user task, each carrying a growing context window, and every single turn re-sends content that has not changed since the conversation started.

This is what engineers mean when they say agentic workflows have a different cost structure. It is not that agents are less efficient than single prompts. It is that a naive agent implementation re-bills the same tokens over and over again across every turn of the loop.

The problem compounds at scale. A customer-facing agent handling 10,000 sessions per day, each running 8 turns, makes 80,000 API calls. If the average turn carries a 4,000-token payload where 2,500 tokens are structural overhead that has not changed, you are billing $150/day on Claude Opus ($15/M input) for tokens you already paid for on the first turn. That is $54,750 per year in avoidable spend on one product, before you account for output tokens or retrieval costs.

This post examines the mechanics of agentic token accumulation, measures where tokens actually go in a representative agent loop, and proposes an optimization framework that addresses each waste source at its source.

Research question

Where do tokens actually accumulate in a ReAct-style agentic workflow, and which optimization techniques reduce cost without meaningfully reducing output quality?

Methodology

We modeled a representative 10-turn research agent with the following characteristics:

  • Task type: multi-step research and summarization (a representative mix of search, read, and synthesis).
  • Tool suite: 10 tools with schemas averaging 350 tokens each (web search, file read, calculator, code execution, and six domain-specific tools).
  • System prompt: 2,000 tokens, fixed across all turns.
  • RAG retrieval: 3 chunks per turn, averaging 800 tokens each, fetched fresh on each turn.
  • Reasoning output: averaging 400 tokens of intermediate reasoning per turn before tool calls.
  • Final synthesis: 1,000-token output on turn 10.

All token counts and costs in this post are illustrative estimates based on these parameters. They are labeled as such throughout. We used published pricing from Anthropic (Claude Opus: $15/M input, $75/M output; Claude Sonnet: $3/M input, $15/M output; Claude Haiku: $0.25/M input, $1.25/M output) to compute cost figures. Actual production numbers will vary with your specific workload.

We compared four agent configurations:

  1. Naive (premium-only): every turn on Claude Opus, no optimization applied.
  2. Router: complexity-based routing per turn; simple retrieval and query generation steps routed to Haiku, synthesis to Opus.
  3. Router + compression: routing plus lossless context compression (JSON minification, schema deduplication, whitespace normalization).
  4. Router + compression + caching: all of the above, with prompt caching on the stable system prompt and tool schemas.

Results: where do tokens go?

The first chart shows the token distribution across a 10-turn agent session. The numbers reveal why agentic costs feel surprising: the content that directly serves the task (search queries, final synthesis) is a small fraction of total token spend.

Figure 1. Illustrative token distribution across a 10-turn ReAct agent loop. The system prompt (20,000 tokens) and tool schemas (12,000 tokens) are repeated on every turn; retrieved context (35,000 tokens) is refreshed each turn. Together they account for roughly 85% of total input tokens.

The key observation: retrieved context alone represents 44% of all input tokens across the session. The system prompt, re-sent on every turn, accounts for another 25%. Tool schemas, also re-sent every turn, add 15%. The actual productive content (search queries and final output) is just 7% of total token spend.

This ratio is the agentic token spiral. The structural overhead grows linearly with the number of turns, while the useful work per turn stays roughly constant. A 10-turn agent carries 10x the system prompt overhead of a single-turn call, even though the system prompt contributed zero new information after turn one.

Results: cost by configuration

The second chart compares estimated monthly cost for 100,000 agent sessions across the four configurations described in the methodology.

Figure 2. Illustrative monthly cost comparison across four agent configurations (100k sessions, avg. 10 turns). Costs based on published Anthropic pricing. Actual savings depend on your turn depth, tool count, and request mix.

A few observations from the cost data:

  • Routing alone cuts cost by 35%. Most intermediate steps (query generation, retrieval scoring, formatting) do not require a frontier model. Routing these to Haiku while preserving Opus for synthesis captures a significant fraction of the savings with zero quality impact on simple steps.
  • Adding compression reduces cost by another 38% on top of routing. Deduplicating tool schemas, minifying JSON payloads, and trimming whitespace are lossless transforms that cut input tokens without altering meaning.
  • Prompt caching on the stable prefix captures the remaining gains. The system prompt and tool schemas (which together represent 40% of input tokens in this model) are stable across turns and across sessions, making them ideal cache targets at Anthropic's 25%-of-normal cache hit rate.

The combined 79% reduction is the result of addressing each waste category at its source, not from any single technique applied in isolation.

Results: latency

Cost is one dimension. Latency is the other. A cheaper agent that takes twice as long is often worse for user experience, and sometimes more expensive in total if it drives retries or session abandonment.

Figure 3. Illustrative median end-to-end latency for a 5-step research agent task across four retrieval strategies. Parallelizing retrieval has the largest single impact on latency.

Sequential search-then-read loops are the dominant latency driver in most research agents. The pattern is: generate a query, call the search tool, receive results, read each result, generate the next query. Each step waits for the previous one. A 5-step loop with 2-second retrieval and 1.5-second LLM calls produces 18-second end-to-end latency for the search and retrieval phase alone.

Parallelizing retrieval cuts this nearly in half. If a step requires three document reads, issuing all three concurrently and waiting for the slowest one is almost always faster than serial execution, even if total token volume is identical.

Selective reading cuts further. Rather than passing all retrieved chunks to the model, a lightweight scoring step (using Haiku, or a small embedding model) can rank chunks by relevance and pass only the top two or three. This reduces both latency (less to process) and cost (fewer input tokens), with acceptable quality impact when the scoring step is tuned.

Results: cost versus quality

The most important chart is the one that shows what you give up as you optimize for cost.

Figure 4. Illustrative cost-quality tradeoff across four strategies. Quality scores based on LLM-as-judge evaluation across diverse agentic tasks. The shaded band marks a representative target quality zone (88-97).

Three findings stand out:

  1. Small-model-first costs more than it appears. Sending every step to the cheapest model reaches the lowest cost ($520/month) but quality drops to 71. Below a certain quality threshold, users retry, abandon the task, or escalate to support. These downstream costs are invisible in the LLM bill but real in the business.
  2. Hybrid routing sits near the quality frontier. Router + compression + caching reaches $890/month with a quality score of 91, compared to $4,200 and 96 for the premium-only baseline. A 5-point quality gap for a 79% cost reduction. Whether that tradeoff is acceptable depends on the task and the quality floor your product requires.
  3. The transition from premium-only to hybrid is not a cliff. Quality degrades gradually as optimization increases, not all at once. This means you can tune the aggressiveness of each technique independently and measure the impact on your specific task distribution.

Token waste breakdown

The fifth chart decomposes avoidable token spend into its specific sources, ordered by contribution. This matters because each waste category has a different fix.

Figure 5. Illustrative breakdown of avoidable token spend by source. Irrelevant retrieved chunks and repeated system prompts together account for half of avoidable waste in this model.

The single largest category of waste is irrelevant retrieved chunks (28%). In most RAG implementations, retrieval returns more context than the model needs. The model pays full price to read all of it. A lightweight re-ranking pass before the main model call can eliminate most of this cost.

The second largest is the repeated system prompt (22%). This is the easiest waste to eliminate: prompt caching on Anthropic's API charges 25% of normal rate for cache hits with a 5-minute TTL. A system prompt that represents 2,000 tokens and is repeated across 10 turns saves 18,000 tokens per session at 75% discount. Over 100,000 sessions, that is 1.8 billion tokens billed at cache hit rates instead of full price.

Redundant tool schemas (18%) are similar: the schemas for tools 1 through 10 are usually identical on turns 2 through 10. Schema deduplication is a lossless transform that can be applied transparently at the proxy layer.

Discussion: what this means for engineering teams

The data above suggests four practical conclusions:

Most agent cost is not in the task, it is in the structure

If your agent makes 10 API calls to complete a task, turns 1 through 9 are carrying overhead that accumulates each turn. The useful signal (the task, the retrieved facts, the reasoning toward an answer) is a minority of total token spend in most production agents. Optimization strategies that target the structural overhead are more impactful than those that target the task content.

Compression and caching are complementary, not competing

Compressing a payload and caching it are different operations. Compression reduces the token count of content that will be billed at full price. Caching reduces the billing rate for content that is stable across requests. Both apply to the system prompt and tool schemas. Applying both together is additive: you pay the cache hit rate on a smaller payload.

Small models can be more expensive than they appear

A Haiku call that produces a wrong or incomplete intermediate result does not just waste the Haiku tokens. It potentially causes the agent to take an extra turn (re-billing the entire context again), or to hallucinate a tool call that returns an error (adding another round trip). Routing should err toward routing up, not down, at steps where errors cascade.

Parallel retrieval does not automatically reduce cost

Parallelizing document reads cuts latency significantly, but it also means all documents are fetched and billed whether or not they turn out to be useful. Pairing parallel retrieval with selective reading (score first, pass only the top chunks) captures the latency benefit without paying for irrelevant context.

Implementation framework

The following seven steps move an unoptimized agent toward the bottom-right quadrant of the cost-quality chart: acceptable quality at substantially lower cost.

  1. Instrument token usage by stage. Before optimizing, measure. Add logging that captures input tokens, output tokens, and cost for each turn of the agent loop, broken down by component (system prompt, tool schemas, retrieved context, user message, prior history). You cannot optimize what you have not measured.
  2. Separate reasoning tokens from context tokens. Reasoning tokens (what the model generates as intermediate steps) are billed at output rates, which are typically 3 to 5 times higher per token than input rates. Short, directed reasoning prompts reduce output token spend. Telling the model to reason in bullet points rather than paragraphs, and to stop when it has enough information, can cut reasoning token spend by 30 to 50%.
  3. Detect over-retrieval and duplicate chunks. Log the relevance scores of retrieved chunks. If your retrieval consistently returns low-relevance documents in the bottom half of the returned set, your top-k parameter is too high. Reducing from k=5 to k=2 on turns where the query is specific can cut retrieved context tokens in half with minimal quality impact.
  4. Route simple steps to cheaper models. Identify which turns in your agent loop are structurally simple: query generation, result formatting, entity extraction, yes/no classification. These steps do not need a frontier model. Routing them to a mid-tier or small model captures most of the routing benefit without touching the steps where quality matters most.
  5. Apply prompt caching to stable prefixes. Mark your system prompt and tool schemas with cache control markers. Move any dynamic content (user IDs, timestamps, session variables) to the end of the prompt, after the static section. Cache hits on a 2,000-token system prompt at Anthropic's 25% cache rate save 1,500 tokens per turn at the cache hit discount. Across a 10-turn session, that is 15,000 tokens saved.
  6. Compress before caching, not instead of it. Context compression (JSON minification, whitespace normalization, schema deduplication) reduces the token count that will be cached and re-billed. Applying compression first and caching second means you cache the compressed payload, which is smaller and therefore cheaper even on cache hits.
  7. Monitor cost, latency, and quality together. Each optimization technique has a quality floor below which it starts producing problems that cost more than it saves. Monitor all three metrics. Set quality thresholds below which routing automatically falls back to a more capable model. Without this, aggressive optimization can produce a system that is cheap to run but expensive to support.

A note on observability

The framework above requires per-turn visibility into what tokens were sent, which model processed them, what the routing decision was, and what the result cost. Most LLM observability setups track total cost per request. They do not break costs down by turn or by component within a turn.

This is the kind of problem that routing and observability layers are designed to solve. Instead of treating every agent turn as an opaque black box, a proxy layer can log the token distribution, routing decision, compression applied, and cache status for each individual API call. This gives engineering teams the data they need to apply the seven steps above based on their actual workload, not on averages from benchmarks like this one.

Nadir provides per-request logging of input tokens, output tokens, routing tier, compression ratio, and cache hit status in both response headers and the dashboard. When you are running an agent that makes 80,000 API calls per day, seeing where the tokens actually went on each call is what turns these framework steps from theory into actionable instrumentation.

Conclusion

The future of LLM cost optimization is not just cheaper models. Cheaper models have been arriving on schedule: Haiku 3.5, GPT-4o mini, Gemini Flash. But agents built without structural awareness of their token usage will consume the savings from cheaper models through increased turn depth, over-retrieval, and repeated overhead. The efficiency gains from better models get captured by inefficient agent architecture.

The frame that matters is not "which model is cheapest?" but "which tokens am I buying that I don't need to buy?" Answering that question requires measurement, then routing, then compression, then caching, applied in that order. Each technique addresses a different waste category. Together they compound rather than add.

The teams that build sustainable AI products are the ones that develop infrastructure to ask this question continuously, not once during a cost optimization sprint. Token spend is a signal. It tells you how your agent is actually working, which steps are expensive relative to their contribution, and where the next optimization is most likely to pay off.
