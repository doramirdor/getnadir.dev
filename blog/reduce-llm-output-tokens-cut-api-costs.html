Reduce LLM output tokens: the other half of your API cost problem

June 22, 2026

Most LLM cost discussions focus on input tokens. That makes sense — they are visible. You can read your prompt, count the context, see the tool schemas stacking up across agent turns. Teams audit their system prompts, compress their JSON payloads, trim their conversation histories, and watch the input token numbers fall.

Then they look at their bill and wonder why it did not fall as much as expected.

The reason is output tokens. On Claude Opus, output tokens cost $75 per million — five times the $15/M input rate. On GPT-4o the ratio is similar. Most production applications generate 200-800 output tokens per request, and most of those applications have never deliberately optimized what the model generates.

Optimizing input tokens and ignoring output is like sealing the leaks in the roof while leaving the windows open. You need both sides.

The pricing asymmetry most teams do not internalize

Here is the full picture across Anthropic's model line, input versus output:

Model Input ($/M tokens) Output ($/M tokens) Output multiplier
Claude Opus 4.6 $15 $75 5×
Claude Sonnet 4.6 $3 $15 5×
Claude Haiku 4.5 $0.80 $4 5×

The multiplier is consistent. Every token the model generates costs five times as much as every token you send. For a typical request with 2,000 input tokens and 400 output tokens, the input costs $0.030 on Opus and the output costs $0.030 as well — equal billing despite a 5:1 token count ratio. Push the output to 800 tokens and output starts dominating the request cost.

Where output tokens come from

A useful way to think about model output is to separate the tokens that deliver value to the user from the tokens that accompany that value. In most production workloads, the ratio is worse than you expect.

Output component Typical token share Eliminable?
Core answer or data 10–30% No — this is what you want
Prose explanation around the answer 20–50% Often yes
Markdown formatting (headers, bullets, bold) 5–15% Yes, for non-rendered contexts
Hedging, caveats, and qualifications 10–25% Often yes
Exposed chain-of-thought reasoning 10–40% Yes, with proper prompt design

For a classification task, a customer support routing decision, a sentiment label, or an entity extraction job, the core answer is often under 20 tokens. The surrounding prose, qualifications, and formatting can push the total to 300+ tokens. Paying $75/M for that overhead adds up fast.

Five patterns that inflate output token counts

Verbose explanations you did not ask for

Frontier models default to explaining their reasoning. Ask "what is the sentiment of this review?" and you get a paragraph. The model identifies positive phrases, notes negative signals, weighs them against each other, and arrives at a conclusion — most of which you do not need if you only want the label.

This is not a model failure. The models are trained to be helpful, and explanation reads as helpful to most users. The fix is not a better model — it is a more specific prompt.

Markdown formatting in non-markdown contexts

When uncertain about rendering context, models default to markdown. Headers, bullet lists, bold text, and fenced code blocks are inserted liberally. If the consumer of the response is an API client parsing JSON, a database write pipeline, or a downstream system that strips formatting, those tokens are pure waste. A 500-token response with heavy markdown might carry 80 tokens of formatting syntax that disappears before any user ever sees it.

Redundant JSON structure

Asking for structured output with verbose schemas generates verbose output. An array of objects with long field names repeated on every element is more expensive than a compact equivalent. The difference matters at scale:

# Verbose schema: ~30 tokens per item
{"sentiment_label": "positive", "confidence_score": 0.92, "reasoning_summary": "..."}

# Compact schema: ~8 tokens per item
{"s": "pos", "c": 0.92}

For a batch job processing 100,000 reviews, the schema choice alone determines whether the output token cost is $0.08 or $0.30 per 1,000 items.

Chain-of-thought exposed in the response

Asking the model to "think step by step" before answering improves accuracy on complex tasks. But if that reasoning appears in the API response and you discard it before returning to the user, you pay full output token rates for tokens nobody reads. A 10-step reasoning chain can run 400-600 tokens before the actual answer appears.

Runaway responses without limits

Without a max_tokens constraint, the model generates as much as the task warrants. An open-ended question about system architecture can produce 2,000 tokens when 300 would serve the user equally well. The model has no incentive to stop early — you do.

Five strategies to reduce output token counts

Use structured outputs (JSON mode)

The highest-impact single change for most applications is switching to constrained structured output. Instead of asking for an answer in prose, define a schema and request only the data you need. The model still reasons internally, but the response is constrained to your format.

Both Anthropic and OpenAI support JSON output mode. Anthropic's tool use API and OpenAI's response_format: {"type": "json_object"} parameter enforce structured responses. The token savings are dramatic for extraction, classification, and decision tasks:

Task type Prose output (tokens) Structured output (tokens) Reduction
Sentiment classification 80–200 8–15 88–93%
Entity extraction (5 fields) 150–400 40–80 70–80%
Routing decision (3 categories) 60–180 5–10 90–95%
Summary generation 100–300 100–300 0% (prose is the deliverable)

Structured output does not help when the deliverable is prose — a user-facing summary, a generated email, an explanation. It helps enormously when the deliverable is data.

Set max_tokens on every request

This is the easiest change and the one most often skipped. Setting a reasonable upper bound per request type prevents runaway outputs and makes your cost envelope predictable.

from openai import OpenAI

client = OpenAI(
    base_url="https://api.getnadir.com/v1",
    api_key="ndr_..."
)

# Classification task: cap at 30 tokens
response = client.chat.completions.create(
    model="auto",
    max_tokens=30,
    messages=[
        {"role": "system", "content": "Classify the ticket. Reply only with the JSON: {\"category\": \"billing|technical|general\"}"},
        {"role": "user", "content": user_ticket}
    ]
)

# Summary task: cap at 200 tokens
response = client.chat.completions.create(
    model="auto",
    max_tokens=200,
    messages=[
        {"role": "system", "content": "Summarize in 2-3 sentences. No markdown."},
        {"role": "user", "content": document}
    ]
)

Set per-task limits, not a single global limit. A routing decision needs 10 tokens. A code review comment needs 300. A full document summary needs 500. Calibrate to actual task requirements, not a blanket cap that truncates legitimate responses.

Tell the model exactly how to respond

Explicit format instructions in the system prompt are more effective than most teams expect. The model does not need to infer what format you want — tell it directly:

  • "Respond with a single word: yes or no."
  • "Reply only with the JSON object. No explanation before or after."
  • "Answer in one sentence. Do not use markdown."
  • "List exactly 3 items. Use plain text, not markdown bullets."
  • "Return the score as a number between 0 and 100. Nothing else."

These instructions work. A system prompt that says "reply only with JSON" dramatically reduces explanatory prose and hedging. The model follows format instructions reliably on frontier models when they are specific and placed in the system prompt rather than the user message.

Separate reasoning from response

If your task requires chain-of-thought accuracy but you do not want the reasoning tokens in your billed output, you have two practical options:

Extended thinking (Anthropic): Anthropic's extended thinking feature runs reasoning in a separate "thinking" block. The model reasons before responding. You pay for thinking tokens at a lower rate and the final response is typically shorter and more accurate than a prompted chain-of-thought response. If accuracy on complex tasks is the goal, this is often cost-neutral or better even though it adds token volume, because the final answer is more concise and correct on the first attempt.

Two-step prompting: Send a reasoning prompt first. Extract the conclusion from the reasoning output. Send only the conclusion to the user. The first call is longer. The second is very short. For tasks where you run many requests and only need the reasoning occasionally (debugging, QA sampling, audit trails), this pattern reduces average output cost significantly.

Route to cheaper models for output-heavy tasks

Cheaper models are not just cheaper per token — they also generate responses with roughly equivalent length to Opus and Sonnet on straightforward tasks. Routing a classification request to Haiku instead of Opus saves 19x on both input and output at the same time.

For tasks where output verbosity matters to quality (complex reasoning, nuanced writing, multi-step analysis), keep the expensive model. For tasks where a shorter, accurate response is the goal, a cheaper model often generates fewer tokens naturally while still being correct.

The routing-output interaction: Nadir's tier-based routing automatically handles this. A request classified as Simple routes to Haiku, which generates the correct 15-token classification response at 1/19th the cost of Opus doing the same thing. The output token savings from routing are automatic — you do not need separate prompt changes for routed requests.

The math at production scale

Take a concrete example: a production customer support triage system running 500,000 requests per month. Each request classifies an incoming ticket into one of six categories. Currently running on Claude Opus with no output constraints, averaging 280 output tokens per response (prose explanation plus the label).

Configuration Output tokens/req Monthly output cost vs. baseline
Baseline (Opus, no constraints) 280 $10,500 —
+ Structured output + max_tokens=20 12 $450 96%
+ Route to Haiku (via Nadir) 12 $24 99.8%

For a classification task, the output token savings from structured output alone are extreme because classification is a fundamentally data-shaped task. Prose output costs $10,500/month for a response format that serves no user — it gets parsed and discarded. Structured output gets the same label in 12 tokens and a fraction of the cost.

Not every workload has this much slack. A customer-facing summarization tool, a code generation assistant, or a writing aid genuinely needs prose output and cannot compress to 12 tokens. But most production LLM applications contain at least some structured, extractive, or classification tasks where the output could be dramatically shorter without any quality loss.

How to audit your output token spend

Every LLM API response includes a usage object with output token counts. Log it. Most teams log the request but not the response metadata, which means they have no visibility into what their output actually costs per endpoint or per task type.

response = client.chat.completions.create(
    model="auto",
    messages=[...]
)

# Log this alongside your request metadata
usage = response.usage
print(f"prompt_tokens: {usage.prompt_tokens}")
print(f"completion_tokens: {usage.completion_tokens}")
print(f"total_tokens: {usage.total_tokens}")

# Attribute to endpoint + model for cost breakdown
log_event({
    "endpoint": "/api/classify",
    "model": response.model,
    "input_tokens": usage.prompt_tokens,
    "output_tokens": usage.completion_tokens,
    "output_cost_usd": usage.completion_tokens / 1_000_000 * output_price_per_million
})

Once you have per-endpoint output token data, the outliers become obvious. A classification endpoint averaging 300 output tokens is a candidate for structured output. An endpoint where output tokens consistently match a natural ceiling (100-150 tokens) may already be well-calibrated. An endpoint with high variance — sometimes 50 tokens, sometimes 800 — is a candidate for explicit max_tokens limits.

Where to start

The order that tends to pay off fastest:

  1. Audit first. Add output token logging to your existing endpoints. Sort by total monthly output tokens. Find the top three endpoints by output cost — those are your targets.
  2. Add max_tokens to every endpoint. Set per-task limits based on what the task actually needs. This takes an hour and immediately caps runaway responses.
  3. Switch classification and extraction tasks to structured output. Any endpoint where you are parsing the model's prose response to extract structured data is a candidate. The savings are typically 80-95% on output tokens for these task types.
  4. Add format instructions to system prompts. "Reply only with JSON. No explanation." on classification and routing tasks. "No markdown formatting." on API-consumed responses. "Answer in two sentences." on summary endpoints with known length requirements.

Combined with input-side optimization from routing, compression, and prompt caching, output optimization closes the loop on both sides of your API cost equation. The teams running LLM-backed products sustainably at scale are the ones who treat both input and output as engineering problems with concrete, measurable solutions.
