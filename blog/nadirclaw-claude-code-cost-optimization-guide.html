Nadir + Claude Code: A Cost Optimization Guide

March 7, 2026

If you use Claude Code for real development work, you have probably noticed the costs. A focused 2-hour session can easily generate 300-500 API calls. Anthropic charges $15/M input tokens and $75/M output tokens for Opus 4.6. On a busy day, that adds up to $10-30 in API costs for a single developer.

The thing is, most of those calls are not doing anything that requires Opus. Claude Code makes a lot of "housekeeping" requests: reading files to understand context, listing directory structures, checking the output of a build command, running grep searches. These requests are simple by nature, and a smaller model handles them identically.

Setup: two minutes, three commands

Nadir runs as a local proxy that intercepts Claude Code's API calls and routes them based on complexity. Here is the full setup:

# 1. Install Nadir
pip install nadirclaw

# 2. Set your API key (Nadir forwards it to Anthropic)
export ANTHROPIC_API_KEY=sk-ant-...

# 3. Start the router
nadirclaw serve

Then tell Claude Code to use the proxy instead of hitting Anthropic directly:

claude config set --global apiBaseUrl http://localhost:8856/v1

That is the entire setup. Claude Code behaves exactly the same. It does not know or care that a proxy is in the middle. Your prompts, your workflows, your key bindings, everything stays identical.

What a real session looks like

Here is a breakdown from an actual 90-minute Claude Code session, refactoring a Node.js backend to add input validation across 12 API endpoints:

Total API calls: 243

  • 147 calls (60%) routed to Haiku ($0.80/M input). File reads, directory listings, package.json checks, grep for existing validation patterns, reading test output. Average ~800 input tokens per call. Cost: ~$0.09
  • 58 calls (24%) routed to Sonnet ($3/M input). Writing individual validation schemas, generating test cases, explaining existing middleware, simple refactors. Average ~2,000 input tokens per call. Cost: ~$0.35
  • 38 calls (16%) stayed on Opus ($15/M input). Planning the validation architecture, writing the shared middleware module, debugging a tricky edge case with nested object validation, reasoning about error handling strategy. Average ~4,000 input tokens per call. Cost: ~$2.28

Total with routing: $2.72

Total without routing (all Opus): $7.30

Savings: 63%

The expensive calls, the ones where Opus actually matters, still go to Opus. The 147 file reads and directory listings that do not need a frontier model get handled by Haiku at a fraction of the cost.

Which Claude Code operations get routed where

To make this more concrete, here are specific Claude Code operations and where Nadir typically routes them:

Haiku (simple tier):

  • Reading file contents (Read tool calls)
  • Listing files with Glob
  • Running Grep searches
  • Checking git status, git diff
  • Formatting or echoing output
  • Simple yes/no or factual questions about the codebase

Sonnet (mid tier):

  • Writing individual functions or methods
  • Generating unit tests for existing code
  • Explaining how a module works
  • Small refactors (rename variable, extract function)
  • Writing docstrings and comments

Opus (complex/reasoning tier):

  • Multi-file refactoring plans
  • Designing new system components
  • Debugging complex issues across multiple files
  • Architectural reasoning ("should we use a queue or polling?")
  • Writing code that requires understanding the full project context

What about quality?

This is the first question everyone asks, and it is fair. The short answer: we have not seen quality degradation on the tasks that matter.

Haiku produces identical output to Opus when you ask it to read a file. There is no "better" way to return file contents. The same goes for directory listings, grep results, and git status output. These tasks have deterministic correct answers, and smaller models handle them fine.

For the mid tier (Sonnet), the quality is very close to Opus on bounded tasks like writing a single function or generating test cases. Sonnet sometimes produces slightly less elegant code, but functionally correct output.

The complex tier stays on Opus specifically because those tasks genuinely benefit from the most capable model. The classifier is intentionally conservative: when it is unsure about a prompt's complexity, it keeps the request on the higher tier. A false positive (cheap model on a hard task) is much worse than a false negative (expensive model on an easy task), so the system errs toward quality.

Cost projections

Based on our data across multiple users and workloads:

  • Light usage (1-2 hours/day of Claude Code): savings of $50-150/month per developer
  • Heavy usage (4-6 hours/day, common for AI-first teams): savings of $200-600/month per developer
  • Team of 5 heavy users: $1,000-3,000/month in reduced API costs

The exact savings depend on your workload mix. Sessions heavy on exploration (lots of file reads, searches) save more than sessions heavy on code generation (more complex-tier calls). Typical savings fall in the 30-60% range.

Optional: add context optimization

If you want to squeeze out additional savings, Nadir also supports context optimization, which compacts the input payload before sending it to the model. Enable it with:

nadirclaw serve --optimize safe

This applies lossless transforms (JSON minification, whitespace normalization, tool schema deduplication) that reduce input token counts by another 20-50% on top of routing savings. See our deep dive on context optimization for the technical details.
